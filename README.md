# Payout Management System (LLD & Implementation)

This repository contains the comprehensive implementation and Low-Level Design (LLD) for a highly resilient **User Payout Management System** handling affiliate sales, advance payouts, and deferred debt recovery.

## 🚀 Tech Stack
- **Backend**: Node.js, Express.js
- **Database**: PostgreSQL with Prisma ORM
- **Frontend**: Next.js (React 19), Tailwind CSS v4, Shadcn UI
- **Infrastructure**: Docker & Docker Compose

---

## 1. Low-Level Design (LLD) Architecture

The system follows a strict **Service-Oriented Architecture (SOA)** using the Controller-Service-Repository pattern. 
- **Controllers** handle HTTP request parsing and response formatting.
- **Services** contain the core domain business logic.
- **Prisma** acts as the Data Access Layer (Repository), ensuring strict ACID compliance via Database Transactions (`$transaction`).

### Payment Gateway Abstraction
To mimic a production environment, the withdrawal flow relies on a mock external Payment Gateway (`paymentGateway.js`). The core `withdrawal.service.js` isolates itself from this external dependency. If a withdrawal fails downstream, the service acts idempotently to refund the user.

## 2. Database Schema & Relationships

The database is normalized and designed for strict financial auditing. We use `BigInt` (paise) for all financial calculations to prevent floating-point precision loss.

### Entities
1. **User**: Represents the affiliate marketer.
2. **Sale**: Represents a product sale generated by the user.
3. **Wallet**: A 1:1 relationship with User, acting as the centralized state of the user's finances.
4. **WalletTransaction**: An immutable append-only ledger tracking every credit/debit to a Wallet.
5. **Withdrawal**: Tracks real-world payout requests.

### Relationships
- `User` (1) — (1) `Wallet`
- `User` (1) — (M) `Sale`
- `User` (1) — (M) `Withdrawal`
- `Wallet` (1) — (M) `WalletTransaction`
- `Sale` (1) — (M) `WalletTransaction`

### Key Schema Decisions
- **`pendingRecoveryPaise` on Wallet**: Instead of allowing the wallet balance to drop into the negative when a sale is rejected, the system records unrecoverable debt in this field. Future payouts are automatically deducted to pay off this debt.
- **`isAdvancePaid` on Sale**: A boolean flag guaranteeing idempotency. A sale can *never* receive double advance payouts.

## 3. Core Business Workflows (Service Layer)

### 3.1 Advance Payout Calculation (`processAdvancePayouts`)
- **Query**: Selects all `PENDING` sales where `isAdvancePaid == false`.
- **Logic**: Calculates 10% of total earnings.
- **Debt Recovery Check**: If the user has a `pendingRecoveryPaise` > 0, the system automatically intercepts the incoming advance and logs a `RECOVERY_DEDUCTION` transaction. Only the remainder is credited to the `balancePaise`.
- **Atomicity**: Updates the `isAdvancePaid` flag and the `Wallet` balance inside a single DB transaction.

### 3.2 Sale Reconciliation (`reconcileSale`)
- **Case 1 (Approved)**: Calculates `Earning - Advance Paid`. Deducts any pending debt (if applicable) and credits the remaining balance.
- **Case 2 (Rejected)**: 
  - Checks if an advance was paid.
  - **Partial Drain**: Drains the wallet balance down to `0`.
  - **Carry Forward**: Any remaining unrecovered debt is appended to `pendingRecoveryPaise`.

### 3.3 Withdrawal Flow (`initiateWithdrawal`)
- Enforces the 24-hour limit using `lastWithdrawalAt`.
- Debits the wallet immediately and creates a `PROCESSING` withdrawal record.
- Dispatches to the external Mock Gateway.
- **On Success**: Finalizes the withdrawal record.
- **On Failure**: Triggers a compensating transaction (`WITHDRAWAL_REFUND`), restoring the wallet balance and **resetting** the `lastWithdrawalAt` lock so the user can immediately retry.

## 5. Edge Cases & Failure Scenarios Handled

1. **Race Conditions & Double Payouts**
   - *Scenario*: Two cron jobs or admins trigger the Advance Payout job at the exact same millisecond.
   - *Solution*: Handled via Prisma `$transaction`. If multiple requests hit the DB, row-level locks prevent the `isAdvancePaid` flag from being evaluated as `false` twice.

2. **Negative Balance Protection (The "Clawback" Edge Case)**
   - *Scenario*: User receives an advance payout of ₹10, withdraws the ₹10 to their bank, and then the sale gets rejected. The system needs to deduct ₹10, but the wallet has ₹0.
   - *Solution*: The system will never result in a `-₹10` balance. Instead, it drains the wallet to `0` and adds `₹10` to `pendingRecoveryPaise` (Debt). The next time the user makes a sale, the system intercepts the payout to recover the debt first.

3. **Mock Gateway Timeouts / Failures**
   - *Scenario*: The bank declines the withdrawal *after* the wallet was already debited.
   - *Solution*: The system safely catches the failure, logs the `failureReason`, writes a `WITHDRAWAL_REFUND` to the ledger (restoring funds), and resets the 24-hour lockout timer so the user isn't punished for a bank failure.

## 6. Key Design Decisions & Trade-Offs

- **No Decimal Types**: Floating point math leads to precision errors in Javascript. Trade-off: Storing everything in `BigInt` as *paise* (cents) requires parsing logic on the frontend, but guarantees 100% financial accuracy.
- **Event-Sourced Ledger vs Mutable Balance**: While the `Wallet` model keeps a mutable `balancePaise` for fast `O(1)` reads, the `WalletTransaction` table acts as an immutable append-only ledger. Trade-off: Uses more storage, but provides a 100% auditable history of exactly where money came from (which is legally required for financial software).
- **Separation of Concerns (Simulated Gateway)**: The core internal database state (debiting the wallet) is resolved *before* calling the gateway, and compensated *after* a gateway failure. This 2-Phase Commit simulation ensures that a hanging network call to a bank doesn't lock the database.
